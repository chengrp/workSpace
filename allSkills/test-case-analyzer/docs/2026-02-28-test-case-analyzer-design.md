## 一、概述

### 1.1 目标

创建一个 Claude Code Skill，能够从需求文档自动生成专业测试用例，支持多种输入/输出格式，并通过学习机制持续进化。

### 1.2 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 需求分析依赖人工经验 | 结构化分析框架 |
| 测试用例编写耗时 | 自动生成 + 模板复用 |
| 格式转换繁琐 | 一键输出多格式 |
| 经验难以传承 | 学习机制持续进化 |

## 二、需求分析方法论

基于视频转录内容，需求分析包含五个核心维度：

### 2.1 核心要素和业务流程

**分析内容**：
- 触发条件：什么事件启动这个流程？
- 计时条件：是否有时间限制？计时起点是什么？计时单位是什么？
- 业务流程：正常流程的完整路径

**关键问题**：
- 计时单位是分钟还是秒？
- 计时实现方式（定时任务轮询 / 消息队列延迟处理）？

### 2.2 判断和执行条件

**分析内容**：
- 执行动作：满足条件后系统做什么？
- 判断时机：系统如何判断条件已满足？
- 处理方式：技术实现方案是什么？

### 2.3 预期结果

**三个维度**：
1. **状态变化**：系统状态如何转变？
2. **数据变更**：哪些数据会改变？
3. **用户感知**：用户会看到/收到什么？

### 2.4 异常流程分析

**三类场景**：

| 类型 | 分析要点 | 示例 |
|------|----------|------|
| **边界值问题** | 极值场景 | 29分59秒支付、30分钟整同时操作 |
| **并发问题** | 同时操作 | 支付和关闭同时发生 |
| **业务交互** | 状态冲突 | 已发货订单不应被关闭 |

### 2.5 外部依赖和配置

**检查项**：
- 是否有可配置参数？
- 是否依赖外部服务？
- 特殊场景处理（服务器重启、网络异常）

**待确认事项**：
- 计时单位
- 实现方式
- 是否可配置
- 并发处理机制
- 状态流转规则

## 三、系统架构

### 3.1 三阶段架构

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  输入处理   │ -> │  分析生成   │ -> │  输出格式化 │
│  Input      │    │  Analysis   │    │  Output     │
└─────────────┘    └─────────────┘    └─────────────┘
       │                  │                  │
       ├─ 多格式支持      ├─ 五维度分析      ├─ Markdown
       ├─ 文档转换        ├─ 分层用例        ├─ Excel
       └─ 智能解析        └─ 完整字段        └─ 反馈收集
```

### 3.2 分析引擎

**输入**：需求文档（文本）

**处理流程**：
```python
1. 文档预处理（去除格式噪声）
2. 关键信息提取（NLP + 规则）
3. 五维度分析
4. 测试用例生成
5. 缺失检测（基于历史模式）
```

**输出**：结构化测试用例数据

## 四、功能设计

### 4.1 输入支持

| 格式 | 处理方式 |
|------|----------|
| Markdown, TXT | 直接读取 |
| PDF, DOCX, PPTX | 调用 all-2md 转换 |
| 网页链接 | 提取内容后分析 |

### 4.2 测试用例结构

**分层结构**：
```
1. 功能测试（正常流程）
2. 边界值测试（极值场景）
3. 状态流转测试（状态机）
4. 回归测试（影响范围）
5. 前端交互测试（UX）
6. 配置测试（可变参数）
```

**测试用例字段**（详细版）：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例ID | TC-{类型}-{序号} | TC-F-001 |
| 标题 | 简洁描述 | 用户下单30分钟后自动关闭 |
| 模块 | 所属功能模块 | 订单管理 |
| 优先级 | P0/P1/P2/P3 | P1 |
| 前置条件 | 执行前状态 | 订单已创建，未支付 |
| 测试步骤 | 详细操作步骤 | 1.创建订单 2.等待30分钟... |
| 预期结果 | 期望输出 | 订单状态变为已关闭 |
| 测试数据 | 具体数据值 | order_id=12345 |
| 状态 | 设计/执行/通过 | 设计 |
| 标签 | 分类标记 | #功能 #自动化 #核心 |

### 4.3 输出格式

**Markdown 结构**：
```markdown
# [需求名称] 测试用例

## 📋 需求分析摘要
- 核心功能：...
- 关键规则：...
- 待确认事项：...

## 🧪 测试用例

### 1. 功能测试
| 用例ID | 标题 | 前置条件 | 步骤 | 预期结果 | 优先级 |
|--------|------|----------|------|----------|--------|
...

### 2. 边界值测试
...

## ❓ 待确认事项
1. 计时单位是分钟还是秒？
2. ...
```

**Excel 结构**：
- **Sheet 1**: 测试用例总表（所有用例）
- **Sheet 2**: 需求分析摘要
- **Sheet 3**: 待确认事项清单
- **Sheet 4**: 测试范围评估

### 4.4 触发方式

| 方式 | 示例 |
|------|------|
| 自然语言 | "分析这个需求并生成测试用例" |
| Slash 命令 | `/test-analyzer` `/需求分析` |

**Node.js**（备用）：
```json
{
  "exceljs": "^4.4.0",
  "marked": "^12.0.0"
}
```

### 7.2 双脚本实现

**优先级**：Python > Node.js

**自动选择逻辑**：
```python
def generate_excel(test_cases, output_path):
    try:
        import openpyxl
        return _generate_with_python(test_cases, output_path)
    except ImportError:
        try:
            return _generate_with_nodejs(test_cases, output_path)
        except:
            raise Exception("请安装 Python openpyxl 或 Node.js exceljs")
```

### 7.3 核心算法

**需求分析算法**：
```python
def analyze_requirement(text):
    # 1. 提取核心要素
    core_elements = extract_core_elements(text)

    # 2. 分析业务流程
    business_flow = analyze_business_flow(text)

    # 3. 识别判断条件
    conditions = extract_conditions(text)

    # 4. 分析异常场景
    exceptions = analyze_exceptions(text)

    # 5. 检查外部依赖
    dependencies = extract_dependencies(text)

    # 6. 加载历史模式
    patterns = load_learned_patterns()

    # 7. 生成测试用例
    test_cases = generate_test_cases(
        core_elements, business_flow, conditions,
        exceptions, dependencies, patterns
    )

    return test_cases
```
