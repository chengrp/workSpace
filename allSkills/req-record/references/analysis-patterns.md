# 需求解码分析技巧指南 (Pro版 - 全模态支持)

本文档详细说明如何从多源异构数据中提炼真实需求，包括文档、音频、视频、手写稿等多模态信息。

---

## 零、多模态预处理 (Multimodal Preprocessing)

### 0.1 证据链等级建立 (Evidence Chain Hierarchy)

在分析前，必须先建立信息源的可靠性等级：

**Tier 等级定义：**

| Tier | 定义 | 典型来源 | 可信度 | 权重 |
|------|------|---------|--------|------|
| **Tier 1** | 最高真理 | 手写批注、白板照片、最新IM聊天记录 | ⭐⭐⭐⭐⭐ | 最高 |
| **Tier 2** | 核心参考 | 会议录音/视频的逐字稿（Transcript） | ⭐⭐⭐⭐ | 高 |
| **Tier 3** | 基础底座 | 正式的招标文件、PRD、历史项目文件 | ⭐⭐⭐ | 中低 |

**关键原则：**
- Tier 1 与 Tier 3 冲突时，**以 Tier 1 为准**
- 例如：标书写预算500万，老板录音说200万 → 以200万为准，并显著高亮该冲突

### 0.2 文件名权重分析

| 文件名特征 | 权重提升 | 理由 |
|-----------|---------|------|
| 包含 "v2", "v3" 等版本号 | +1 | 表示迭代更新 |
| 包含 "老板修改版", "领导批示" | +2 | 直接体现决策者意志 |
| 包含 "final", "最终版" | +1 | 但需警惕"最终版_v2"的情况 |
| 包含日期（如20241015） | +1 | 可判断时效性 |
| 包含 "草稿", "Draft" | -1 | 可能不是最终版本 |

### 0.3 多模态格式归一化

**处理流程：**

1. **检测非文本文件**：
   - 音频 (MP3, M4A, WAV) → 调用转录工具获取逐字稿
   - 视频 (MP4, AVI, MOV) → 提取音频后转录
   - 图片 (JPG, PNG) → 调用OCR工具提取文字
   - 手写稿 → OCR + 手写识别

2. **质量自检**：
   - 转录中出现 "[unintelligible]" → 标记为"低置信度"
   - OCR识别率 < 80% → 标记为"需人工复核"
   - 手写字迹潦草 → 在"待确认清单"中列出

3. **来源标注**：
   - 所有提取的信息必须标注来源索引
   - 格式：`[文件名] [时间戳/位置]`
   - 示例：`[meeting_01.mp3] [04:12]` 或 `[scan_01.jpg] [右上角]`

## 一、全模态交叉验证 (Multimodal Cross-Validation)

### 1.1 听"弦外之音" (For Audio/Video Transcripts)

**说话人区分 (Speaker Diarization)：**

| 说话人类型 | 优先级 | 关注重点 | 处理策略 |
|-----------|-------|---------|---------|
| **大老板/决策人** | P0 | 方向性、预算、验收标准 | 原话引用，在方案首页体现 |
| **技术负责人** | P1 | 技术细节、集成接口 | 详细记录技术约束 |
| **具体经办人** | P2 | 执行细节、操作流程 | 作为补充参考 |
| **顾问/第三方** | P3 | 可能带有倾向性 | 需验证其动机 |

**迟疑标记 (Hesitation Markers)：**

| 迟疑信号 | 含义 | 处理方式 |
|---------|------|---------|
| "原则上..." | 必有例外 | 标记"需确认例外场景" |
| "大概..." | 不确定 | 归类为模糊信息 |
| "这个嘛..." | 有顾虑 | 可能是真实担忧点 |
| "到时候再说" | 拖延策略 | 标记"大概率不实施" |

**语调分析：**

| 语调信号 | 推断 | 应对策略 |
|---------|------|---------|
| 提及某竞品时语气不耐烦 | 负面合作经验 | 方案中避免类似功能 |
| 某个话题声音突然变大 | 特别重视/情绪激动 | 设为 P0 优先级 |
| 长时间沉默/支支吾吾 | 可能有隐瞒或不确定 | 标记"需深入确认" |

### 1.2 看"笔尖流露" (For Handwritten/Images)

**手写信号解读：**

| 手写特征 | 含义 | 处理策略 |
|---------|------|---------|
| **圈出/高亮部分** | 绝对重点 | 设为 P0 |
| **边缘潦草笔记** | 真实担忧 | 在待确认清单中列出 |
| **打叉/划掉** | 明确排斥或取消 | 不在方案中体现 |
| **问号/感叹号** | 疑惑或强调 | 需澄清或重点关注 |
| **多次修改痕迹** | 反复斟酌 | 可能是争议点 |

**白板/照片信号：**

| 信号 | 含义 | 处理策略 |
|------|------|---------|
| 流程图/架构图 | 客户心中的技术蓝图 | 方案需对齐 |
| 手写关键词 | 最核心的关注点 | 在方案中突出 |
| 多次擦除重写 | 争议点或不确定点 | 需特别关注 |

### 1.3 冲突仲裁 (Conflict Resolution)

**仲裁原则：**

| 冲突场景 | 仲裁规则 | 高亮方式 |
|---------|---------|---------|
| Tier 1 vs Tier 3 | **以 Tier 1 为准** | ⚠️ 冲突警告 |
| 说话人A vs 说话人B | 以职级高者为准 | 标注双方观点 |
| 正式文档 vs IM聊天 | 以最新时间的为准 | 标注时间差异 |
| 标书 vs 实际沟通 | 以实际沟通为准 | 在方案中解释差异 |

**冲突处理模板：**
> ⚠️ **冲突警告**：标书显示预算300万，但聊天记录（2024-10-15）透露今年砍了预算。**以200万为准**，需在方案中提前说明范围调整。

## 二、显性门槛识别 (Hard Constraints Detection)

### 1.1 预算相关信号词

| 信号词 | 含义 | 提取要点 |
|-------|------|---------|
| "预算不超过XX" | 硬性上限 | 记录具体数字 |
| "投资额约XX" | 软性参考 | 标记为"待确认" |
| "成本控制在" | 约束条件 | 提取百分比或绝对值 |
| "性价比" | 模糊指标 | 需进一步澄清 |

### 1.2 周期相关信号词

| 信号词 | 含义 | 提取要点 |
|-------|------|---------|
| "XX前上线" | 硬性deadline | 记录具体日期 |
| "分X期实施" | 阶段性交付 | 拆分每期时间节点 |
| "响应时间不超过X" | 性能指标 | 归类到技术约束 |
| "快速部署" | 模糊表述 | 标记"需澄清具体时间" |

### 1.3 资质相关信号词

| 信号词 | 含义 | 提取要点 | 潜规则关联 |
|-------|------|---------|-----------|
| "ISO 27001" | 信息安全认证 | 记录证书级别 | - |
| "等保X级" | 网络安全等级 | 记录具体级别 | - |
| "国产化" | 自主可控要求 | 明确是芯片/OS/数据库哪一层 | 触发"自主可控"潜规则 |
| "信创" | 信息技术应用创新 | 需明确具体产品清单 | 触发"自主可控"潜规则 |
| "涉密" | 保密资质要求 | 确认具体保密级别 | - |
| "自主可控" | 国产化替代硬要求 | 排除纯进口产品 | **SC敏感词：国产化适配硬要求** |

### 1.4 技术约束信号词

| 信号词 | 含义 | 提取要点 |
|-------|------|---------|
| "基于XX开发" | 技术栈锁定 | 记录具体技术 |
| "兼容XX系统" | 集成要求 | 明确集成接口标准 |
| "私有部署" | 部署方式 | 与SaaS互斥 |
| "云原生" | 架构偏好 | 可能暗示容器化、微服务 |
| "支持XX协议" | 通信标准 | 记录协议版本 |

## 三、隐性倾向挖掘 (Hidden Tendencies Detection)

### 2.1 技术偏好推断

#### 判断维度
| 暗示信号 | 可能倾向 | 验证方法 |
|---------|---------|---------|
| 频繁提及"微服务""容器化" | 云原生架构 | 检查是否提及K8s/Docker |
| 强调"数据安全""自主可控" | 私有部署/国产化 | 查找敏感行业关键词（政府/金融） |
| 多次出现某竞品术语 | 已有产品依赖 | 搜索竞品专有名词 |
| 强调"快速迭代""灵活扩展" | 敏捷开发偏好 | 检查DevOps工具提及 |
| 提及"统一管理""集中管控" | 集中式架构 | 与分布式架构信号对比 |

#### 确信度评估标准
- **高确信度**：同一倾向在3处以上明确提及，或使用排他性表述（"必须是""只能"）
- **中确信度**：同一倾向在2处提及，或使用倾向性表述（"建议""优先"）
- **低确信度**：仅1处模糊提及，需二次确认

### 2.2 决策关注点推断

#### 关注点分类
| 关注点类型 | 关键词特征 | 决策者动机 | 潜规则关联 |
|-----------|-----------|-----------|-----------|
| **安全合规** | "合规""审计""安全""风险""监管" | 怕担责，规避风险 | - |
| **创新亮点** | "领先""创新""标杆""示范" | 想升职，追求政绩 | - |
| **成本控制** | "降本""增效""性价比" | 预算压力，KPI导向 | - |
| **用户体验** | "易用""友好""满意度" | 面向最终用户 | - |
| **技术先进** | "前沿""最新""技术架构" | 技术背景决策者 | - |
| **高层关注** | "领导关注""主要领导""集团要求" | 最高优先级，没得商量 | **SC敏感词：必须在方案首页体现** |
| **稳健优先** | "成熟案例""有经验""验证过" | 极度保守，不想做小白鼠 | **SC敏感词：方案侧重稳定性，避免激进创新** |

#### 交叉验证
- 如果"安全合规"与"创新亮点"同时出现，通常前者优先级更高
- 政府项目：安全合规 > 创新 > 成本
- 互联网公司：创新 > 用户体验 > 成本
- 传统企业：成本 > 安全 > 创新关注点

### 2.3 竞品暗示识别

| 暗示方式 | 示例 | 推断 |
|---------|------|------|
| 直接提及 | "类似Salesforce的功能" | 已有Salesforce使用经验 |
| 术语借用 | "需要Story、Epic管理" | 可能参考Jira/敏捷开发工具 |
| 功能对标 | "要像XX那样一键发布" | 期望对标竞品体验 |
| 隐性对比 | "不要像XX那样复杂" | 负面竞品经验 |

### 2.4 SC 敏感词库（潜规则解码）

> 这些词汇是资深架构师在实战中总结的"行话密码"，看似普通，实则暗藏玄机。

| 敏感词 | 表面含义 | 深层潜规则 | 应对策略 |
|-------|---------|-----------|---------|
| **"自主可控"** | 技术自主性 | **国产化适配硬要求** | 纯进口产品直接排除，需明确芯片/OS/数据库/中间件的国产化替代方案 |
| **"不仅限于"** | 示例性列举 | **需求蔓延红色预警** | 此处是范围边界模糊点，必须在方案中明确设置"范围红线"，防止后期无休止加需求 |
| **"成熟案例"** | 要求有经验 | **客户极度保守，怕做小白鼠** | 方案必须侧重稳定性、可验证性，避免激进创新；重点展示同类行业成功案例 |
| **"领导关注"** | 重要性提示 | **最高优先级，没得商量** | 必须在方案首页/首屏突出体现，且需提供可直接汇报的高层摘要材料 |
| **"后续考虑"** | 暂不实施 | **大概率永远不会实施** | 可当作"口惠而实不至"，不需要为此预留过多架构空间 |
| **"原则上"** | 通用规则 | **必有例外，需追问** | 后面通常跟着"但是XX情况除外"，需主动询问例外场景 |
| **"酌情处理"** | 灵活掌握 | **责任转移，需谨慎** | 意味着没有明确标准，出了问题是你判断的锅，建议要求书面确认 |
| **"尽量"** | 努力做到 | **做不到也正常** | 非刚性约束，资源冲突时可优先放弃 |
| **"参考"** | 借鉴标准 | **非强制，可偏离** | 有选择权，可根据实际情况调整 |
| **"同步"** | 一并考虑 | **时间上可能有先后** | 可能是"也要做"的委婉说法，不一定严格同时进行 |

#### 潜规则使用原则

1. **敏感词优先级高于字面意思**：看到"自主可控"时，第一反应是国产化，而不是"技术自主"
2. **潜规则触发特殊处理**：
   - "不仅限于" → 立即记录到风险点的"信息模糊"类别
   - "成熟案例" → 在方案中增加"案例对标"章节
   - "领导关注" → 调整为 P0 级目标
3. **多敏感词叠加效应**：
   - "成熟案例" + "自主可控" = 政府/国企项目，极度保守
   - "不仅限于" + "酌情处理" = 需求模糊度高，需大量澄清

## 四、噪音过滤 (Noise Filtering)

### 3.1 官话套话识别

| 噪音类型 | 典型表述 | 处理方式 | 潜规则关联 |
|---------|---------|---------|-----------|
| 宏大叙事 | "打造行业标杆""建设一流体系" | 忽略，除非有具体指标 | - |
| 同义反复 | "提高效率""优化流程" | 忽略，除非有可量化指标 | - |
| 空洞描述 | "灵活配置""强大扩展" | 标记"需澄清具体场景" | - |
| 装饰性表述 | "业界领先""国际先进" | 忽略，不包含技术约束 | - |
| 范围模糊 | "功能不仅限于XX""包括但不限于" | **红色预警**：需求蔓延风险 | **SC敏感词：需设置范围红线** |
| 责任转移 | "酌情处理""根据实际情况" | 警惕：无明确标准，风险转嫁 | **SC敏感词：需书面确认** |
| 非刚性表述 | "尽量""参考""原则上" | 可作为资源冲突时的放弃项 | **SC敏感词：非硬性约束** |

### 3.2 信息提取优先级

```
硬性约束 > 具体功能描述 > 量化指标 > 模糊表述 > 官话套话
```

### 3.3 过滤流程

1. **第一遍扫描**：标注所有数字、日期、专有名词、标准名称
2. **第二遍扫描**：标注所有"必须""应该""要求"等义务性表述
3. **第三遍扫描**：剔除无修饰词的形容词、副词
4. **第四遍整合**：将分散信息归类到对应约束类别

## 五、文档类型适配 (Document Type Adaptation)

### 4.1 招标文件特征

| 特征 | 处理要点 |
|------|---------|
| 结构化程度高 | 优先阅读"技术要求""评分标准"章节 |
| 包含评分细则 | 高分项=显性门槛，其他项=隐性倾向 |
| 有附件清单 | 检查附件中的详细规格说明 |
| 格式规范 | 参数表、功能表通常是最可靠信息源 |

### 4.2 会议纪要特征

| 特征 | 处理要点 |
|------|---------|
| 口语化表达 | 需推断技术含义 |
| 信息分散 | 需跨段落整合关联信息 |
| 有讨论过程 | 争议点=关键风险点 |
| 可能存在遗漏 | 标记"建议录音核对" |

### 5.3 IM聊天记录特征

| 特征 | 处理要点 |
|------|---------|
| 高度碎片化 | 需按时间线重构需求演化 |
| 上下文依赖强 | 注意"上文中提到的X"类指代 |
| 表情符号语气 | 可辅助判断倾向性 |
| 可能包含情绪 | 情绪强烈点=重点关注点 |

### 5.4 多模态组合场景

| 组合场景 | 处理策略 |
|---------|---------|
| 标书 + 会议录音 | 以录音中"老板发言"为准 |
| 标书 + 手写批注 | 批注内容=最新变动，优先级最高 |
| 聊天记录 + 会议纪要 | 以时间最新的为准，注意"说一套写一套" |

## 六、常见陷阱 (Common Pitfalls)

### 5.1 需避免的错误

| 错误类型 | 表现 | 后果 |
|---------|------|------|
| **过度解读** | 从模糊表述推断过多 | 误判客户需求 |
| **忽视语境** | 割裂提取单句 | 错失依赖关系 |
| **遗漏约束** | 忽略"隐性"但关键的表述 | 方案不满足要求 |
| **混淆优先级** | 所有信息等同对待 | 资源分配错位 |
| **证据不足** | 推断缺乏原文引用 | 无法向客户验证 |

### 5.2 质检检查清单

分析完成后，自查：
- [ ] 每个显性门槛都有原文依据
- [ ] 每个隐性倾向都标注了确信度
- [ ] 所有推断都有证据引用
- [ ] 模糊信息都标记了"需确认"
- [ ] 矛盾点都识别并记录
- [ ] 官话套话已过滤

## 七、实战示例

### 示例 1：基础解码

**原文：**
> "系统应采用云原生架构，确保高可用性。预算控制在200万以内，需在Q4前上线。要求符合等保2.0标准，并优先考虑国产化解决方案。"

**解码输出：**

| 类别 | 提取内容 | 证据 |
|------|---------|------|
| 技术约束 | 云原生架构 | "采用云原生架构" |
| 预算 | 200万上限 | "预算控制在200万以内" |
| 周期 | Q4前上线 | "需在Q4前上线" |
| 合规要求 | 等保2.0 | "符合等保2.0标准" |
| 隐性倾向 | 国产化优先（中确信度） | "优先考虑" |
| 模糊点 | "高可用性"无具体指标 | 标记需确认SLA |

### 示例 2：SC 敏感词解码

**原文：**
> "本项目要求系统自主可控，功能不仅限于用户管理，还需要有成熟案例支撑。领导关注数据安全，要求尽量支持等保三级。"
