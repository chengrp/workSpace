# 四支柱知识框架 (Core Pillars)

构建强大提示词的四个核心支柱：角色、逻辑、语料、格式。

## 1. 角色 (Role)

**定义**: AI应该扮演什么身份？具备什么专业能力？

### 设计要点

- **专业身份**: 明确具体的专业角色（如"资深产品经理"、"Python架构师"）
- **能力特质**: 列举3-5个核心能力（如"善于沟通"、"逻辑缜密"）
- **视角立场**: 从什么角度思考问题（如"用户视角"、"商业视角"）

### 示例

```
你将扮演一名资深UX设计师，具备以下特质：
- 10年+互联网产品设计经验
- 擅长用户调研和可用性测试
- 熟悉无障碍设计标准(WCAG)
- 注重数据驱动决策
```

---

## 2. 逻辑 (Logic)

**定义**: 工作流程、步骤、推理链条是什么？

### 设计要点

- **工作流程**: 清晰的步骤序列（Step 1, Step 2, Step 3...）
- **决策规则**: 什么情况下做什么选择
- **约束条件**: 绝对不能做什么
- **优先级**: 多目标时的权衡顺序

### 结构模式

**线性流程**: 适用于固定任务
```
1. 分析输入
2. 执行处理
3. 生成输出
```

**条件分支**: 适用于复杂场景
```
IF 条件A THEN 执行X
ELSE IF 条件B THEN 执行Y
ELSE 执行Z
```

**迭代循环**: 适用于优化类任务
```
WHILE 未满足条件:
    尝试改进
    评估结果
```

---

## 3. 语料 (Context/Data)

**定义**: 需要什么背景知识、数据、参考资料？

### 设计要点

- **领域知识**: 行业术语、概念框架
- **参考标准**: 格式规范、质量标准
- **样例数据**: Few-shot examples
- **约束清单**: 必须遵守的规则列表

### 注入方式

**直接嵌入**: 适用于稳定内容
```markdown
## 知识库
以下术语定义：
- 术语A: ...
- 术语B: ...
```

**引用外部文件**: 适用于大型内容库
```markdown
## 参考文档
详见 [references/domain-knowledge.md]
```

---

## 4. 格式 (Format)

**定义**: 输入输出格式、结构化要求、模板规范？

### 设计要点

- **输入格式**: 用户应提供什么结构的输入
- **输出格式**: 期望的输出结构（Markdown/JSON/表格等）
- **模板约束**: 必须包含的章节、字段
- **样式要求**: 语气、风格、语言

### 常用输出格式

**结构化文本**: Markdown标题层级
```markdown
# 主标题
## 副标题
### 小标题
```

**结构化数据**: JSON/YAML
```json
{
  "field1": "value1",
  "field2": "value2"
}
```

**表格数据**: Markdown表格
```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| ... | ... | ... |
```

## 四支柱组合模式

### 高精度任务模式
- **Role**: 专家级身份
- **Logic**: 多步骤验证流程
- **Context**: 详细领域知识
- **Format**: 严格结构化输出

### 创意生成模式
- **Role**: 创意型角色
- **Logic**: 发散→收敛流程
- **Context**: 灵感参考库
- **Format**: 灵活格式

### 对话助手模式
- **Role**: 服务型角色
- **Logic**: 状态机对话流
- **Context**: 知识库+用户历史
- **Format**: 自然对话

---

## 四支柱检查清单

创建提示词时，确保每个支柱都有清晰定义：

- [ ] **角色**: AI的身份是否明确？能力是否具体？
- [ ] **逻辑**: 工作流程是否完整？边界条件是否清晰？
- [ ] **语料**: 所需知识是否完备？样例是否充分？
- [ ] **格式**: I/O契约是否明确？模板是否可执行？
