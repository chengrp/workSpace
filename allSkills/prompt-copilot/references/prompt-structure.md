# 提示词结构规范 (Prompt Structure Specification)

本文档详细说明了最终提示词的标准Markdown结构模板及各部分编写指南。

## 完整模板

```markdown
# 提示词名称: [由用户或你共同命名]
# 版本: [例如: V1.0, V1.1, V2.0]
# 作者: [用户的名字] & Prompt Co-Pilot

---

## 🎯 1. 核心目标 (Objective)

*   这个提示词旨在通过将[明确的输入]转化为[明确的输出]，为[目标用户]提供[核心价值]。

**编写指南:**
- 用一句话描述提示词的使命
- 明确输入→输出的转化关系
- 说明为目标用户带来的价值
- 避免技术术语，用业务语言描述

**示例:**
> 这个提示词旨在通过将[用户需求描述]转化为[结构化PRD文档]，为[产品经理]提供[快速输出高质量需求文档的能力]。

---

## 📝 2. 核心I/O契约 (Core I/O Contract)

*   **输入 (Input):** [此处详细描述已确认的输入格式和内容]
*   **输出 (Output):** [此处详细描述已确认的输出格式和内容]

**编写指南:**

### 输入部分
- 明确用户应该提供什么
- 如果有格式要求，给出示例或模板
- 说明可选字段 vs 必填字段

**示例:**
```markdown
**输入 (Input):**
用户将提供一个JSON对象，包含以下字段：
- `user_story`: 用户故事描述（必填，1-3句话）
- `acceptance_criteria`: 验收标准（可选，列表形式）
- `priority`: 优先级（可选，P0/P1/P2/P3）
- `context`: 背景信息（可选，自由文本）
```

### 输出部分
- 明确AI应该返回什么
- 给出输出结构示例
- 说明质量标准

**示例:**
```markdown
**输出 (Output):**
一份完整的PRD文档，包含：
1. 需求概述（1-2段）
2. 功能详细描述（分点列出）
3. 用户故事与验收标准（按角色组织）
4. 边界情况说明（什么不做）
5. 技术考虑点（可选）

输出格式：Markdown，使用标准标题层级（H1-H3）
```

---

## 🎭 3. 角色 (Role)

*   为了最好地完成上述契约，你将扮演一名[角色名称]，具备[专业能力1]、[专业能力2]的特点。

**编写指南:**
- 角色要具体，避免泛泛而谈（不要说"AI助手"，要说"资深产品经理"）
- 列举3-5个核心能力特质
- 可以提到经验年限、专长领域

**示例:**
```markdown
你将扮演一名资深产品经理，具备以下特质：
- 8年+互联网产品经验，熟悉B2B SaaS产品
- 擅长将模糊需求转化为清晰的产品规格
- 精通敏捷开发和用户故事拆解
- 注重用户体验和可执行性
```

---

## ⚙️ 4. 工作流程与规则 (Workflow & Rules)

*   **工作流程:**
    1.  [步骤1]
    2.  [步骤2]
    3.  ...

*   **核心规则:**
    *   [规则1：必须做什么]
    *   [规则2：绝对不能做什么]
    *   [规则3：风格、语气等要求]

**编写指南:**

### 工作流程
- 使用编号列表，步骤清晰
- 每个步骤描述具体行动
- 复杂任务可以分组（如：准备阶段→执行阶段→输出阶段）

**示例:**
```markdown
**工作流程:**
1. **分析阶段**: 仔细阅读用户故事，识别核心诉求
2. **补充阶段**: 如果信息不足，主动提出澄清问题
3. **结构化阶段**: 按照PRD模板组织信息
4. **优化阶段**: 检查输出的完整性和可读性
5. **输出阶段**: 生成最终的Markdown文档
```

### 核心规则
- 用"必须"、"禁止"、"应当"等词明确要求
- 区分必须遵守的规则 vs 建议性指导

**示例:**
```markdown
**核心规则:**
1. **必须**在信息不足时主动提出澄清问题，而不是猜测
2. **禁止**编造不存在的功能或需求
3. **输出必须**包含"边界情况"章节，明确说明什么不在范围内
4. 语气应当**专业但不僵化**，可以适当使用例子说明
5. 如果用户提供了技术术语，**保持原样**不要替换
```

## 💡 5. 关键知识/语料 (Knowledge & Context)

*   你的所有回答都必须严格基于以下提供的背景知识和专业术语：
*   [此处用于注入用户提供的关键信息]

**编写指南:**
- 提供领域特定的术语定义
- 列出参考标准或框架
- 注入Few-shot examples（可选）
- 如果内容较多，可以引用外部文件

**示例:**
```markdown
你的所有回答都必须严格基于以下背景知识：

**术语定义:**
- **用户故事**: 描述用户价值的简短陈述，格式为"作为[角色]，我想要[功能]，以便[价值]"
- **验收标准**: 判定功能完成与否的具体条件
- **技术债务**: 为了短期速度而牺牲的代码质量，需要后续偿还

**参考框架:**
- INVEST原则（用户故事质量标准）
- MoSCoW优先级分级法

**常见问题类型:**
- 功能型需求（系统应该做什么）
- 非功能型需求（性能、安全、可用性等）
- 约束条件（技术、业务、法律限制）
```

## 📐 7. 输出格式 (Output Format)

*   你的所有输出都必须遵循[格式要求，例如：Markdown, JSON等]。
*   [如果需要，提供一个输出模板或结构示例]。

**编写指南:**
- 明确指定输出格式（Markdown/JSON/表格等）
- 如果是结构化输出，提供模板
- 说明章节顺序和标题层级

## 版本管理规范

### 版本号规则
- **V1.0**: 初始版本
- **V1.1**: 小优化（修正错误、完善说明）
- **V2.0**: 重大改动（结构变化、新增核心功能）

### 版本历史示例
```markdown
# 版本历史
- V1.0 (2024-01-15): 初始版本，基础PRD生成功能
- V1.1 (2024-01-20): 增加技术考虑点章节，优化输出模板
- V2.0 (2024-02-01): 新增多用户故事支持，重构工作流程
```

## 质量检查清单

完成提示词后，逐项检查：

### 结构完整性
- [ ] 包含全部7个核心章节
- [ ] 版本号和作者信息完整
- [ ] Markdown格式正确

### 内容质量
- [ ] 核心目标清晰具体
- [ ] I/O契约明确无歧义
- [ ] 角色定义具体专业
- [ ] 工作流程可执行
- [ ] 核心规则可验证
- [ ] 示例真实有代表性
- [ ] 输出格式可复现

### 可用性
- [ ] 新手可以直接使用
- [ ] 不需要额外解释
- [ ] 边界情况已考虑
- [ ] 有明确的验证标准
