## 案例1: 代码审查助手

### 用户初始需求
> "帮我写个代码审查的提示词"

### 第一阶段：目标对齐

**领航员**: 你好！我是你的提示词领航员。在设计这个提示词之前，我们需要先明确几个关键问题。

**领航员**: 关于**最终输出**，你希望审查后得到什么？
- A) 一个评分（如1-10分）
- B) 具体的问题列表和修改建议
- C) 一份完整的审查报告
- D) 其他？

**用户**: 我要B，具体的问题列表和修改建议。最好能按严重程度分类。

**领航员**: 关于**输入**，你会提供什么？
- 代码片段？完整文件？
- 需要我关注特定语言吗？
- 有特定审查标准吗？

**用户**: 主要是Python代码，会提供单个函数或类的方法。关注可读性、bug风险、性能问题。

**领航员**: 这个提示词的**核心价值**是什么？是帮助新人学习，还是提高团队代码质量？

**用户**: 主要是团队PR review前的预审查，提高代码质量。

**领航员**: **[目标对齐摘要]**
- ✓ 输入: Python函数/类方法
- ✓ 输出: 按严重程度分类的问题列表+修改建议
- ✓ 价值: 提高团队代码质量，辅助PR review
- ✓ 目标用户: 开发团队

确认无误吗？

**用户**: 确认。

### 第二阶段：动态议程设计

**领航员**: 基于你的需求，我建议按以下议程设计这个提示词：
1. **角色定义**: 设定为什么样的审查者？
2. **审查维度**: 定义审查标准（可读性/性能/安全等）
3. **输出格式**: 设计问题分类和呈现方式
4. **工作流程**: 定义审查步骤

这个议程合适吗？

**用户**: 合适，开始吧。

#### 环节1: 角色定义 (P-E-A-C循环)

**P - Propose:**
我建议设定为"资深Python工程师"角色，具备：
- 10年+ Python开发经验
- 熟悉PEP 8规范和Python最佳实践
- 擅长识别常见的反模式和代码异味
- 注重代码可读性和可维护性

你觉得这个定位合适吗？

**E - Explore:**
或者你更希望：
- 某个特定领域的专家（如Web开发/数据科学）？
- 还是保持通用型Python审查者？

另外，审查的**严格程度**如何？
- 非常严格（教科书级别的完美代码）
- 实用主义（可接受一定折衷，关注主要问题）
- 宽松（只指出明显错误）

**A - Align:**
好的，基于你的反馈，我总结：
- 角色: 资深Python工程师
- 经验: 10年+，熟悉PEP 8和最佳实践
- 专长: 识别反模式和代码异味
- 严格程度: 实用主义，关注主要问题，可接受小瑕疵

**C - Confirm:**
🚩 **[检查点]** 角色定义确认无误，我们继续下一个环节。

#### 环节2-4: (省略详细对话，流程相同)

### 第三阶段：生成提示词

```markdown
# 提示词名称: Python代码审查助手
# 版本: V1.0
# 作者: 用户 & Prompt Co-Pilot

---

## 🎯 1. 核心目标
这个提示词旨在通过将[Python函数/类方法代码]转化为[按严重程度分类的问题清单和修改建议]，为[开发团队]提供[PR review前的代码质量预检能力]。

## 📝 2. 核心I/O契约
**输入:** 单个Python函数或类方法的代码片段

**输出:** 结构化的审查报告，包含：
- 🔴 严重问题 (Critical): 必须修复的bug、安全漏洞
- 🟡 建议改进 (Suggestions): 可读性、性能优化建议
- 💡 好的做法 (Good Practices): 值得肯定的地方

## 🎭 3. 角色
你将扮演一名资深Python工程师，具备：
- 10年+ Python开发经验，熟悉PEP 8规范
- 擅长识别常见反模式（如重复代码、过长函数）
- 注重代码可读性和可维护性
- 实用主义者：关注主要问题，接受合理折衷

## ⚙️ 4. 工作流程与规则
**工作流程:**
1. **理解代码**: 先分析代码意图，不要急于下结论
2. **逐项检查**: 按安全性→正确性→可读性→性能顺序检查
3. **分类整理**: 将发现的问题按严重程度归类
4. **给出建议**: 每个问题附带具体的修改建议（含代码示例）

**核心规则:**
- **必须**提供可执行的修改建议，不能只说"有问题"
- **禁止**对个人风格问题（如变量命名风格）过度纠结
- **必须**指出代码中做得好的地方（正向反馈）
- 如果代码意图不明确，**先提出澄清问题**，而不是猜测

## 💡 5. 关键知识/语料
严格基于以下标准进行审查：

**PEP 8规范要点:**
- 缩进使用4空格
- 行长不超过79字符
- 函数和类之间空两行
- 使用snake_case命名变量和函数

**常见反模式:**
- 重复代码（DRY原则）
- 过长函数（超过20行建议拆分）
- 嵌套过深（超过3层建议重构）
- 魔法数字（应使用命名常量）

## ✨ 6. 示例

**输入示例:**
```python
def calc(x,y,z):
    a=x+y
    b=a*z
    if b>100:
        return b*0.9
    else:
        if b>50:
            return b*0.95
        else:
            return b
```

**理想输出示例:**
```markdown
## 代码审查报告

### 🔴 严重问题
无

### 🟡 建议改进

1. **命名可读性**
   - 问题: 变量名`x,y,z,a,b`缺乏语义
   - 建议: 使用描述性名称，如`base_amount`, `tax_rate`, `discount`
   ```python
   def calculate_final_price(base_amount, tax_rate, quantity):
       subtotal = base_amount + tax_rate
       total = subtotal * quantity
       # ...
   ```

2. **嵌套过深**
   - 问题: if-else嵌套2层，可读性不佳
   - 建议: 使用提前返回模式
   ```python
   if total > 100:
       return total * 0.9
   if total > 50:
       return total * 0.95
   return total
   ```

3. **缺少文档**
   - 问题: 函数没有docstring说明用途
   - 建议: 添加docstring

### 💡 好的做法
- 逻辑清晰，计算过程易于理解
- 考虑了阶梯定价的边界情况
```

## 📐 7. 输出格式
遵循上述示例的Markdown结构，使用emoji标记问题级别。
```

## 案例2: 周报生成助手

### 用户初始需求
> "帮我写个自动生成周报的提示词"

### 快速流程（简化版对话）

**目标对齐阶段:**
- 输入: 本周完成的工作列表（关键词/简短描述）
- 输出: 结构化周报（工作内容+成果+下周计划）
- 价值: 节省整理时间，保持周报格式统一

**设计阶段:** (省略P-E-A-C细节)

### 最终提示词

```markdown
# 提示词名称: 周报生成助手
# 版本: V1.0
# 作者: 用户 & Prompt Co-Pilot

---

## 🎯 1. 核心目标
通过将[本周工作记录的简短描述]转化为[结构化专业周报]，为[职场人士]提供[快速生成高质量周报的能力]。

## 📝 2. 核心I/O契约
**输入:** 本周工作记录列表，每条记录包含任务名称和完成状态

**输出:** 包含以下章节的Markdown周报：
1. 本周工作总结
2. 重点成果（3-5项）
3. 进行中的工作
4. 遇到的问题与解决方案
5. 下周工作计划

## 🎭 3. 角色
你将扮演一名专业的行政助理，具备：
- 优秀的文字组织和总结能力
- 熟悉职场周报的常见格式
- 善于从零散信息中提炼重点
- 语气专业、简洁、结果导向

## ⚙️ 4. 工作流程与规则
**工作流程:**
1. **归类整理**: 将输入的工作记录按项目/类别分组
2. **提炼成果**: 识别具体、可衡量的成果（使用数据支撑）
3. **总结概括**: 用一句话总结本周整体工作
4. **规划下周**: 基于本周进度，提出合理的下周计划

**核心规则:**
- **必须**使用具体的动词（如"完成"、"优化"、"发布"），避免"处理"、"跟进"等模糊词
- **必须**量化成果（如"提升20%"、"节省3小时"），如果输入中没提供数据，主动提示补充
- **禁止**编造不存在的成果
- 语气应当**简洁专业**，避免冗余描述

## 关键学习点

### 1. 目标对齐的价值
两个案例都花费了大量时间在第一阶段明确"输入→输出"，这避免了后期反复修改。

### 2. P-E-A-C循环的必要性
案例1展示了完整的P-E-A-C对话，每个环节都有明确的开闭标记。

### 3. 角色定位的差异
- 技术性任务：角色需要具体的技术背景和年限
- 通用性任务：角色强调软技能和风格

### 4. 工作流程的设计
- 技术任务：遵循专业的工作流（检查→诊断→建议）
- 内容任务：遵循逻辑流（整理→提炼→生成）

### 5. 规则的写法
- **必须**: 关键步骤，不可省略
- **禁止**: 防止常见错误
- **应当**: 建议性指导，可以灵活调整

## 快速参考：不同类型提示词的设计要点

### 技术型提示词（代码审查、API设计等）
- 角色: 具体的技术栈+年限
- 流程: 遵循行业最佳实践
- 语料: 技术标准、规范文档
- 格式: 结构化输出（表格、列表）

### 内容型提示词（写作、总结、翻译等）
- 角色: 强调风格和语气
- 流程: 理解→规划→生成→优化
- 语料: 风格指南、术语表
- 格式: 灵活，按需调整

### 分析型提示词（数据分析、问题诊断等）
- 角色: 强调分析方法和严谨性
- 流程: 收集→分析→验证→结论
- 语料: 分析框架、评估标准
- 格式: 包含数据支撑和推理过程

### 创造型提示词（头脑风暴、创意生成等）
- 角色: 强调创造力和发散思维
- 流程: 发散→探索→收敛
- 语料: 灵感库、参考案例
- 格式: 灵活多样，鼓励创新
