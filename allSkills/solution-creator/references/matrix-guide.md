# 产品矩阵填写指南

> 本文档帮助你理解如何正确填写 `product-matrix.md`，这是方案生成的核心依据。

## 核心原则

### 原则 1：真实准确
- **绝对禁止**夸大产品功能或性能指标
- 所有数据必须可验证、可交付
- 如实标注实施依赖，不可遗漏

### 原则 2：价值导向
- 不是"卖功能"，而是"卖价值"
- 每个功能都要回答：客户为什么需要？

### 原则 3：差异化清晰
- 明确对比竞品的优势
- 避免使用"业界领先"、"国内首创"等模糊表述
- 用具体的技术特点或性能指标说话

## 各列填写详解

### 功能模块 (Feature)

**命名建议**：
- 使用行业通用术语 + 产品特色
- 例如："智能知识库"、"自动巡检 Agent"、"容量预测引擎"

**结构建议**：
- 可按产品线分层级
- 同一模块的多个子功能可用"|"分隔

### 对应痛点 (Pain Point)

**描写公式**：
> 在 [具体场景] 下，客户面临 [具体问题]，导致 [负面后果]

**示例**：
- ❌ "检索效率低"（太笼统）
- ✅ "运维人员在处理故障时，需要在 20+ 个文档中手动查找信息，平均耗时 2 小时，且经常遗漏关键信息"

**量化要求**：
- 尽可能量化问题的严重性
- 使用时间、成本、错误率等具体指标

### 核心价值主张 (Value Proposition)

**转化公式**：
> 从 [现状] 变为 [目标状态]，实现 [量化价值]

**示例**：
- ❌ "提供智能搜索，提高效率"（没有量化）
- ✅ "从"人找人"变为"人找知识"，检索效率提升 70%，新人培训周期缩短 50%"

**价值类型**：
1. **效率提升**：时间缩短 X%、处理量提升 X 倍
2. **成本降低**：人力成本降低 X 万元/年、资源利用率提升 X%
3. **质量改善**：错误率降至 X%、准确率提升至 X%
4. **风险规避**：提前 X 小时预警、减少 X% 的 downtime

### 竞品差异化 (Differentiator)

**差异化维度**：

1. **技术架构**
   - 私有化部署 vs 公有云
   - 自研算法 vs 开源方案
   - 实时处理 vs 批处理

2. **性能指标**
   - 响应速度：毫秒级 vs 秒级
   - 并发量：支持 10x 并发
   - 准确率：提升 20 个百分点

3. **成本优势**
   - 不限 Token，无额外成本
   - 按需付费，降低 50% 成本
   - 一次投入，终身使用

4. **合规性**
   - 数据完全自主可控
   - 通过等保三级认证
   - 支持国产化适配

5. **独特能力**
   - 独有的预测算法（专利号：XXXXX）
   - 业界唯一的 XX 能力
   - 自研的 XX 引擎

**避免使用**：
- ❌ "业界领先"（什么领先？）
- ❌ "国内首创"（有证据吗？）
- ❌ "性能优异"（多优异？）

### 实施依赖 (Implementation Dependency)

**依赖类型**：

1. **技术依赖**
   - 需要 GPU 服务器（推荐配置：NVIDIA A100 × 2）
   - 需要兼容 Linux 内核 5.0+
   - 需要支持 Docker/Kubernetes 环境

2. **数据准备**
   - 需提供近 6 个月的告警历史数据
   - 需提供清洗后的用户行为数据
   - 需要按照指定格式整理知识文档

3. **环境要求**
   - 需要内网隔离环境
   - 需要开放特定端口的网络权限
   - 需要与现有 SSO 系统对接

4. **人力投入**
   - 需要配置 1 名专职管理员
   - 需要对运维团队进行 3 天培训
   - 需要业务部门配合梳理流程

5. **第三方依赖**
   - 需对接 Prometheus API
   - 需购买阿里云 OSS 服务
   - 需申请 OpenAI API Key

## 常见错误

### 错误 1：功能描述过于技术化
- ❌ "基于 BERT 的语义向量化检索"
- ✅ "智能理解用户意图，支持自然语言搜索"

### 错误 2：价值主张没有量化
- ❌ "显著提高效率"
- ✅ "检索效率提升 70%，平均响应时间从 2 小时降至 5 分钟"

### 错误 3：差异化不清晰
- ❌ "性能业界领先"
- ✅ "响应时间 < 100ms，比竞品快 5 倍"

### 错误 4：实施依赖不完整
- ❌ "需要服务器"
- ✅ "需要 GPU 服务器（推荐：NVIDIA A100 × 2，内存 100G+）"

### 错误 5：痛点描述模糊
- ❌ "用户体验不好"
- ✅ "用户完成一次操作需要点击 8 次，平均耗时 3 分钟，流失率高达 40%"

## 维护建议

1. **定期更新**
   - 每季度至少更新一次性能指标
   - 新功能上线后及时补充

2. **版本管理**
   - 保留历史版本，方便追溯
   - 标注最后更新日期

3. **评审机制**
   - 技术团队评审：确保功能描述准确
   - 产品团队评审：确保价值主张合理
   - 销售团队评审：确保差异化有说服力

**记住**：产品矩阵的质量直接决定生成方案的质量。请务必认真填写！
